# `rope` — synthesizable VecU rotary-position-embedding slice (RTL)

**Status:** RTL complete + micro-sequenced, **bit-exact to the RoPE LUT golden**
(`rtl/tb/tb_rope.sv`: 23 committed corner rows + 300-row random stress, **0
mismatches**), **Yosys-clean** (436 FFs at HEAD_DIM=8/MAX_POS=16, **no latches** —
`t:$dlatch = 0` asserted). **Hardened to GDSII on GF180 + Sky130** (LibreLane 3.0.5,
clean sign-off — see below).
**Home:** `rtl/rope.sv` (+ `rtl/tb/tb_rope.sv`, `rtl/tb/gen_rope_vectors.py`, golden
`sw/reference_model/rope_ref.py`).
**One line:** applies rotary position embedding to a decode Q or K vector — the raw
Q/K path the chip-top needs before Q·Kᵀ (the cosim's loaded Qwen tiles are already
RoPE'd, so this is for the from-scratch attention path).

## What it computes (the hardware algorithm — not an exact rotation)

For a head-dim-`H` fp16 vector at sequence position `pos`, it rotates each
(even, odd) channel pair `i = 0..H/2-1`:

```
x'_2i   = x_2i * cos(theta_i * pos) - x_2i+1 * sin(theta_i * pos)
x'_2i+1 = x_2i * sin(theta_i * pos) + x_2i+1 * cos(theta_i * pos)
theta_i = BASE^(-2i/H)          (BASE = 10000, the Llama/Qwen RoPE base)
```

- **sin/cos LUT (the codebook ROM RoPE table).** A ROM indexed by
  `(pos, channel-pair)` holds `COS[pos][i] = round_fp16(cos(pos*theta_i))` and
  `SIN[pos][i] = round_fp16(sin(pos*theta_i))`. The fp16 rounding of cos/sin **is**
  the LUT approximation this block models. Default geometry: `HEAD_DIM=8`
  (4 pairs), `MAX_POS=16` → a 64-entry cos + 64-entry sin ROM (embedded case,
  generated by `rope_ref.py --emit-sv`).
- **fp32 datapath, one fp16 rounding.** Each product is an fp32 multiply and each
  pair-sum an fp32 add/sub, rounded **once** to fp16 on emit — the same IEEE
  primitives (`fp16↔fp32`, `fp32_add/sub/mul`, `fp32_to_fp16`) as
  `mate_pv_fp16` / `vecu_softmax`. So the error is the cos/sin LUT quantization,
  not fp16 product drift.

## Accuracy (the two bars, kept separate)

| bar | what it proves | result |
|---|---|---|
| **(a) RTL bit-exact to the LUT-golden** | the RTL matches its spec exactly | `make sim_rope`: 23 corner rows + 300 random, **0 mismatches** |
| **(b) LUT-golden vs exact fp64 rotation** | the LUT is accurate enough | **max rel-err ≈ 7.4e-4** to the vector peak; cos/sin fp16-quant alone ≤ **2.4e-4**. Run `python sw/reference_model/rope_ref.py`. |

Well within the `rel_err < 5e-3` bar. `pos = 0` is an exact identity (cos=1, sin=0).

## Interface (house streaming style)

- **LOAD** — one fp16 channel per clock on `s_data`, `s_valid=1`, `s_last=1` on the
  final channel (`HEAD_DIM` total). `pos` is sampled on the **first** channel (it is
  constant for the vector).
- **EMIT** — after the micro-sequence finishes, the block streams the `HEAD_DIM`
  rotated channels on `y_data`/`y_valid`, `y_last` on the final one. The consumer
  waits on the `y_valid` handshake, so the (data-independent) latency is transparent.
- `busy` high from the first channel until the last rotated channel.

## Timing structure — micro-sequenced

Channels are buffered on load, then per pair a short micro-sequence runs at most
**one fp32 op per register-to-register path** (a few independent ops in parallel per
cycle, like `vecu_softmax`), so the GF180 ss corner closes with **normal** resizing:

| step | op(s) |
|---|---|
| READ  | widen x0, x1, cos, sin |
| MUL1  | `pa = x0*cos`, `pb = x1*sin` |
| MUL2  | `pc = x0*sin`, `pd = x1*cos`; `y0 = pa-pb` |
| ADD   | `y1 = pc+pd` |
| EMIT0 | `y_data = fp16(y0)` |
| EMIT1 | `y_data = fp16(y1)`; advance pair |

6 cycles/pair → 24 cycles for a HEAD_DIM=8 vector. Op sequence is fixed, so the
result is bit-exact to `rope_ref.py`.

## Synthesis

Yosys `synth_rope.ys` (`make synth_rope`): **436 FFs** (HEAD_DIM=8/MAX_POS=16), **no
latches** (`select -assert-count 0 t:$dlatch`), ~44.9k NAND2-equivalent cells
(abc `-fast -g NAND` lower bound). Function locals are default-initialised so no
transient latch inference either.

## Scope / still open

- Default geometry `HEAD_DIM=8`, `MAX_POS=16`. A production head dim (e.g. 64/128)
  and longer context widen the ROM; regenerate the LUT (`rope_ref.py --emit-sv`,
  bump `HEAD_DIM`/`MAX_POS`) — for a large position range the ROM should become a
  real SRAM/codebook lane rather than an embedded case.
- One vector at a time (single Q or K head); batching across heads is future work.
